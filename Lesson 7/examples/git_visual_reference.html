The Git Workflow — Four Areas

Every file in Git lives in one of four places. Understanding this is the key to everything.

Area 1
📝
Working Directory
Your files as you edit them. Changes here are not yet tracked by Git.
Area 2
🧺
Staging Area
A "draft zone". You choose which changes to include in the next snapshot.
Area 3
💾
Local Repository
Your permanent save history. Each commit is a snapshot you can return to.
Area 4
☁️
Remote (GitHub)
The online backup. Shared with collaborators. Accessed via push & pull.
📝 Working Dir ── git add ──► 🧺 Staging ── git commit ──► 💾 Local Repo ── git push ──► ☁️ GitHub
☁️ GitHub ── git pull ──► 💾 Local Repo + 📝 Working Dir
💡 Video Game Analogy

Think of each commit as a save point in a video game. If you break something later, you can reload from any previous save. Unlike overwriting a file, Git keeps every save point forever, and you can jump back to any of them.

💡 Shopping Basket Analogy (Staging Area)

The staging area is like a supermarket basket. You browse the shop (edit files), put items in the basket (git add), and when you're ready, you go to the checkout (git commit). You decide exactly what goes into each basket — and therefore each snapshot.

Core Git Commands

Click any command block to copy it to your clipboard.

git init
Initialise a new Git repository in the current folder. Creates a hidden .git/ directory — Git's brain.
git init
$ git init
Initialized empty Git repository in ./my-project/.git/
⚠️ Always run inside your project folder, not on the Desktop.
git status
Show the current state of all files — untracked, modified, staged. Your most-used command.
git status
$ git status
On branch main
Untracked files:
index.html
Changes to be committed:
new file: style.css
git add
Move changes into the staging area. Stage one file, multiple files, or everything at once.
git add index.html
git add .
# git add . stages everything in the
# current folder and subfolders
git commit -m "..."
Save all staged changes as a permanent snapshot with a descriptive message.
git commit -m "Initial commit - Add HTML structure"
$ git commit -m "Add nav bar"
[main a3f9c12] Add nav bar
1 file changed, 22 insertions(+)
git log
View the full commit history. Use --oneline for a compact, one-line-per-commit view.
git log --oneline
$ git log --oneline
a3f9c12
(HEAD → main) Fix broken nav link
b0e7d23
Add hero section
3c1a5f8
Initial commit
git diff
Show exactly what lines changed in your working directory since the last commit. Lines with + were added; - were removed.
git diff
$ git diff
@@ -1,4 +1,5 @@
- <title>My Site</title>
+ <title>My Portfolio</title>
git remote add origin
Link your local repository to a remote repository on GitHub. origin is the conventional name for your primary remote.
git remote add origin https://github.com/username/repo.git
git remote -v
git push
Upload your local commits to GitHub. Use -u origin main for the first push to set the default upstream.
git push -u origin main
git push
# -u sets the default so future
# pushes only need: git push
git pull
Fetch changes from GitHub and merge them into your local repo. Run this at the start of every work session.
git pull
$ git pull
remote: Counting objects: 3
Updating b0e7d23..a3f9c12
index.html | 2 +-
git branch & git config
One-time setup and branch utilities. Configure your identity and rename branches.
git config --global user.name "Your Name"
git config --global user.email "you@email.com"
git branch -M main

Understanding git status Output

The output of git status changes as you move files between areas.

State 1 — Untracked
Untracked files:
index.html
style.css
Git sees these files but is NOT tracking them. Run git add to start tracking.
State 2 — Staged
Changes to be committed:
new file: index.html
new file: style.css
Files are in the staging area. Run git commit to save the snapshot.
State 3 — Clean
On branch main
nothing to commit,
working tree clean
All changes are committed. Your working directory matches the last snapshot.
State 4 — Modified (unstaged)
Changes not staged:
modified: index.html
You've edited a tracked file. Run git add index.html to stage the changes.
State 5 — Partially Staged
Changes to be committed:
modified: index.html
Changes not staged:
modified: style.css
One file staged, one not. This is why staging gives you fine-grained control.
State 6 — Deleted File
Changes not staged:
deleted: about.html
A tracked file was deleted. Run git add about.html then commit to record the deletion.

Connecting to GitHub — Step by Step

The full workflow for taking a local project live on GitHub.

#StepCommand / ActionNotes
1 Initialise local repo git init Run once inside your project folder
2 Stage all files git add . The dot = everything in current folder
3 First commit git commit -m "Initial commit" You must have at least one commit before pushing
4 Create GitHub repo github.com → New repository Do NOT add README/gitignore — leave it empty
5 Link remote git remote add origin <URL> Copy the HTTPS URL from GitHub
6 Rename branch git branch -M main GitHub uses main by default
7 First push git push -u origin main The -u sets the default; future pushes just need git push
✓ Verify git remote -v Should show your GitHub URL for fetch and push
❌
Common mistake: Ticking "Add a README file" when creating the GitHub repo. If your local project already has files, this creates a conflict and your first git push will be rejected. Always create an empty repo when pushing an existing project.
🔑
Authentication: GitHub no longer accepts your account password for git push. You need a Personal Access Token (PAT). Go to: GitHub → Settings → Developer settings → Personal access tokens → Generate new token. Use this token wherever Git asks for a password.

.gitignore — Telling Git What to Ignore

Some files should never be committed. The .gitignore file tells Git to skip them.

Why do I need a .gitignore?

Some files are auto-generated, system-specific, or contain sensitive data (passwords, API keys). Committing them clutters your history and can be a security risk. A .gitignore file tells Git to pretend these files don't exist.

Common files to ignore

PatternWhat it ignores
.DS_StoremacOS system files
Thumbs.dbWindows thumbnail cache
node_modules/npm dependencies (huge!)
.envEnvironment variables / secrets
*.logAll log files
dist/Build output folders
.vscode/VS Code personal settings

Pattern syntax

.gitignore
# Lines starting with # are comments
 
# Ignore a specific file
.env
 
# Ignore all files with this extension
*.log
 
# Ignore an entire folder
node_modules/
 
# Ignore in all subfolders
**/.DS_Store
 
# Un-ignore a file (exception)
!important.log
💡
Create a .gitignore file in the root of your project (same level as your index.html). Add it and commit it as part of your initial commit. The file has no extension — it's literally just called .gitignore.
# Starter .gitignore for a web project
.DS_Store
Thumbs.db
*.log
.env
node_modules/

Writing Good Commit Messages

A good commit message is one of the most valuable habits you can build as a developer.

✅ Good examples

"Initial commit - Add portfolio HTML structure"
"Add responsive navigation bar"
"Fix broken image link on projects page"
"Update footer with social media links"
"Style hero section with gradient background"
"Add README with project description"

❌ Bad examples

"stuff"
"changes"
"fix"
"final"
"v2"
"asdfjkl"
"updated things"
💡 The Rule

Write your message as if completing the sentence: "This commit will..."
→ "This commit will Fix broken image link on projects page"
