Branching, Merging & PRs

A visual guide covering parallel development, resolving merge conflicts, and professional GitHub workflows.

1

The Branching Concept

Why do we use branches? To work parallel without breaking the main codebase.

💡
The Golden Rule: The main branch should always be stable and deployable. All direct development happens on Feature Branches. Once a feature is complete and verified, it is merged back into `main`.
2

Core Branching Commands

git branch
List all local branches. The active one is marked with an asterisk (*).
git branch
$ git branch
feature/about-page
* main
git checkout -b
Shortcut: Create a new branch AND switch to it immediately. Often use `git switch -c` in modern Git.
git checkout -b feature/login
$ git checkout -b feature/login
Switched to a new branch 'feature/login'
git checkout
Switch between existing branches. Your working files will update immediately to match the chosen branch.
git checkout main
git branch -d
Delete a local branch. Use this when the feature is merged into `main`.
git branch -d feature/login

Merge Types

Combining the history of two branches.

A. Fast-Forward Merge
If `main` hasn't had any new commits since the feature branch diverged, Git simply moves the `main` pointer forward.
# 1. Be on main
git checkout main
# 2. Merge feature branch
git merge feature/add-footer
Updating a1e8f23..c92a1b5
Fast-forward
index.html | 2 ++
B. Default (3-Way) Merge
If `main` HAS moved forward (e.g., someone else merged a feature), Git combines them via a "Merge Commit".
Merge made by the 'ort' strategy.
style.css | 4 +++...

Resolving Merge Conflicts

Conflicts happen when you and a teammate edit the exact same lines.

Don't panic! It's normal. Git is just asking you to make a decision.
index.html (Conflict Markers)
Some surrounding code...

<<<<<<< HEAD (Your current branch)
<h1>Alex Chen | Web Developer</h1>
======= (Separator)
<h1>Alex Chen — Portfolio</h1>
>>>>>>> feature/new-heading (Incoming branch)

More surrounding code...

How to Fix:

  1. Open the conflicting file in your editor (VS Code).
  2. Find the conflict markers (search `<<<<`).
  3. Decide which version to keep, or manually combine them.
  4. Delete all line markers (`<<<<`, `====`, `>>>>`).
  5. Save the file.
  6. Run `git add ` and `git commit` to finalize.

Pull Requests (GitHub Workflow)

The standard way professional teams review and combine code.

#StepCommand / ActionNotes
1 Create Feature Branch git checkout -b feat/new-page Isolate your work locally.
2 Commit Work git add .
git commit -m "..."		 Make small, logical commits.
3 Push Branch git push origin feat/new-page Uploads the branch to GitHub.
4 Open PR github.com → "Compare & pull request" Base: `main` ← Compare: `feat/new-page`
5 Code Review Discuss on GitHub Teammates approve or request changes.
6 Merge "Merge pull request" button Happens entirely on the GitHub server.
7 Sync Local repo git checkout main
git pull		 Crucial! Get the newly merged code locally.

Branching Cheat Sheet

🌲 Local Branches

git branch
List branches
git checkout -b <name>
Create & switch instantly
git checkout <name>
Switch to branch
git branch -d <name>
Delete a merged branch
git branch -D <name>
Force delete branch

🔀 Merging & Remote

git merge <name>
Merge <name> into current branch
git push origin <name>
Push branch to GitHub
git push origin --delete <...>
Delete branch on GitHub
git fetch --prune
Clear stale remote branches locally
